1. What This Is

The Human Trafficking Law & Policy Monitor is a living, searchable database tracking federal and state legislation, civil and criminal litigation, and peer-reviewed research related to human trafficking. It is designed to support researchers, policy analysts, and advocacy practitioners who need a unified view of the legal and evidentiary landscape across these three domains -- a task that normally requires navigating Congress.gov, PACER, academic databases, and federal agency portals separately.

The tracker combines a manually curated seed dataset of 257 entries with daily automated fetches from four external sources. All data is stored as plain JSON and served as a static website -- there is no backend server, no login, and no tracking. The application runs entirely in the user's browser.

The three categories covered are:

  • Legislation -- federal bills, enacted laws, and state statutes related to human trafficking, including companion bills and authorization acts.
  • Regulatory -- agency rules, proposed rules, guidance documents, and financial crime advisories from federal agencies (classified separately from legislation to reflect their distinct procedural pathway and enforceability).
  • Lawsuits -- civil suits and criminal cases, with particular attention to civil actions brought under 18 U.S.C. 1595, the beneficiary liability provision of the Trafficking Victims Protection Act (TVPA), and hotel/hospitality industry litigation.
  • Research -- peer-reviewed journal articles, systematic reviews, government-funded studies, and policy reports relevant to anti-trafficking intervention, survivor outcomes, and program evaluation.

2. Technical Architecture

The entire application is a single HTML file (index.html) with embedded CSS and JavaScript. There is no build step, no framework dependency, and no server-side code. Data is loaded at startup via the browser's fetch() API from four JSON files in the data/ directory:

FileContentsUpdated by
data/legislation.jsonLegislation and regulatory entriesDaily auto-fetch + manual
data/lawsuits.jsonCivil and criminal casesManual + community submissions
data/research.jsonResearch papers and reportsDaily auto-fetch (Overton) + manual
data/meta.jsonEntry counts and last-updated timestampsRewritten on every data write

The site is hosted on GitHub Pages, which serves the static files directly from the repository. All automated data updates are handled by a GitHub Actions workflow that runs daily at 8:00 AM EST, commits updated JSON files to the repository, and triggers a Pages rebuild.

User annotations (stars, notes) are stored exclusively in the browser's localStorage API under the key namespace ht-tracker-*. They are never transmitted to any server.

3. Data Sources

The tracker draws from six sources. Four are automated; two are manual-only.

Congress.gov API

What it provides: Federal bills, amendments, cosponsor counts, and the most recent floor or committee action for any bill updated within the lookback window.

Access: Free API key available at api.congress.gov. Configured via the CONGRESS_API_KEY GitHub Actions secret.

How it is queried: The fetch script sends separate search requests for seven distinct keyword phrases (e.g., "human trafficking," "uyghur forced labor," "survivor services trafficking"). Up to 20 results per query are returned, sorted by most-recently-updated. Each result is then checked against a 22-term keyword list applied to the bill title before being included. This double-filter reduces false positives from broadly-worded search results.

Fields populated automatically: title, bill identifier, status (mapped from action text), sponsor, cosponsor count, introduced date, latest action date and text, URL, and updates timeline entry.

Federal Register API

What it provides: Final rules, proposed rules, notices, and presidential documents published by federal agencies within the lookback window.

Access: No key required. Publicly available REST API at federalregister.gov.

How it is queried: A single search using the combined query "human trafficking OR forced labor OR trafficking victims," requesting documents published on or after the lookback date, limited to 40 results sorted by newest. Results are filtered by the same 22-term keyword list applied to title plus abstract. All documents retrieved this way are classified as type regulatory, not legislation.

Fields populated automatically: title, document number, publication date, issuing agency names, document type, abstract (truncated to 400 characters for the summary field), and HTML URL.

FinCEN RSS Feeds

What it provides: Press releases and advisories from the Financial Crimes Enforcement Network related to anti-money-laundering requirements, suspicious activity reporting (SAR), and financial typologies associated with trafficking.

Access: No key required. Two public RSS feeds are monitored: news releases and advisories/bulletins/fact sheets.

How it is queried: Both feeds are parsed on each fetch run. Items are filtered by a separate 8-term keyword list that includes "trafficking," "forced labor," "AML," "SAR," and "modern slavery." Items older than the lookback window are skipped by comparing publication date against the lookback cutoff. All FinCEN items are classified as type regulatory.

Overton

What it provides: Peer-reviewed research papers, policy reports, and grey literature indexed from academic journals and policy sources, with metadata including DOI, authors, journal, and abstract.

Access: Institutional subscription required. Overton is a research intelligence platform; access is through an institutional Bearer token configured as the OVERTON_API_KEY GitHub Actions secret. If the key is absent, the research fetch is skipped gracefully and existing research entries are preserved.

How it is queried: Six search queries (e.g., "human trafficking intervention evaluation," "sex trafficking survivors long-term outcomes") are sent to the Overton search API, requesting papers published on or after the lookback date, 20 results per query. A deduplication set prevents the same paper from being added twice across multiple queries. Results are verified against the 22-term keyword list on title plus abstract before inclusion.

Fields populated automatically: title, DOI, authors (up to 6), journal, publication date, and abstract (as summary). Fields such as evidence_strength, outcome_frame, methodology, and population are left blank on auto-fetch and must be manually curated.

PACER / Federal Court Dockets

What it provides: Federal civil and criminal case filings, docket entries, and court orders.

Access: No public API suitable for automated retrieval. All lawsuit entries are added manually by the research team after review of PACER or public case documents.

State Legislatures

What it provides: State-level bills, enacted statutes, and regulations related to human trafficking.

Access: No uniform national API exists for state legislative data. Entries are added manually. LegiScan provides a multi-state API but is not currently integrated.

4. How Automated Fetching Works

The fetch workflow runs daily at 8:00 AM Eastern via GitHub Actions, using a 90-day rolling lookback across all sources. It can also be triggered manually from the Actions tab. The core design principle is non-destructive merging: automated fetches never overwrite manually curated fields. Only status, latest_action, latest_action_date, and cosponsors are auto-updated. The updates timeline is append-only, with deduplication keyed on date and event text. All other fields -- including summary, description, outcome_frame, and evidence_strength -- are never touched. If no changes are found, no commit is created.

6. Classification Logic & Assumptions

Several classification decisions were made in building this dataset. These are documented here so that users can apply appropriate interpretive caution.

Trafficking Type Inference

For auto-fetched legislation and research entries, trafficking type (sex, labor, or both) is inferred from keywords in the title and abstract. The inference function checks for sex-trafficking-associated terms ("sex trafficking," "prostitution," "csec," "commercial sex," "escort") and labor-trafficking-associated terms ("labor trafficking," "forced labor," "debt bondage," "supply chain").

Bills or papers that mention neither set of terms explicitly are classified as ["sex", "labor"] by default. This is a conservative fallback -- general anti-trafficking legislation (e.g., TVPA reauthorizations, victim services funding bills) typically addresses both forms. Users who need precise type attribution should verify the classification against the source document and correct it manually in the data file.

Status Mapping from Congress.gov

Congress.gov returns the text of the most recent floor or committee action rather than a standardized status code. The fetch script maps this to one of six status values using keyword matching on the action text:

Detected phraseAssigned status
"signed by president" or "became public law"Enacted
"passed senate" or "passed/agreed to in senate"Passed Senate
"passed house" or "passed/agreed to in house"Passed House
"committee" or "subcommittee" or "hearing"In Committee
(no match)Introduced

This mapping is intentionally conservative and will not capture all possible legislative states. In particular, it does not distinguish between bills that died in committee and bills that were actively marked up, or between Senate and House versions of a companion bill. For legislative tracking that requires this granularity, the item's linked Congress.gov page should be consulted directly.

Outcome Frame Classification

The outcome_frame field describes how a law, regulation, or study frames its theory of success -- that is, what it treats as a meaningful result. This field was added to support research into the gap between service-delivery accountability (counting beds, intakes, referrals) and longer-term survivor well-being. It is not a standard field in any external API and must be manually assigned for all entries. Values used in the dataset:

  • Service Metrics Only -- success is defined by service counts, throughput, or process compliance with no survivor-reported outcomes.
  • Survivor: Short-Term (0-6 months) -- measures survivor outcomes at less than six months post-intervention.
  • Survivor: Medium-Term (6-12 months)
  • Survivor: Long-Term (>12 months)
  • Mixed / Both -- combines service metrics and survivor outcomes.

For auto-fetched research entries from Overton, outcome_frame is left blank. It is populated manually as papers are reviewed by the research team.

Evidence Strength

The evidence_strength field is applied to research entries and reflects an assessment of methodological rigor. It is manually assigned and not derived from any automated scoring. Values are: strong, moderate, limited, and descriptive (for qualitative or non-causal work that is valuable but not designed to establish efficacy).

Case Category

Lawsuit entries are tagged with a case_category reflecting the industry or institutional sector named as defendant. Categories currently used include: HOSPITALITY INDUSTRY, ONLINE PLATFORMS, FINANCIAL (banks and payment processors), LABOR / EMPLOYMENT, GOVERNMENT, and INDIVIDUAL. These are intended to enable filtering by sector rather than to imply legal determinations about the defendant.

Policy Mechanism

Legislation entries may carry a policy_mechanism tag describing the primary instrument the law employs. This field is manually curated. Examples: Service Funding, Criminalization, Platform Liability, Supply Chain, Victim Identification, Survivor Compensation, Cross-Sector Coordination. A single bill may address multiple mechanisms but only one primary value is stored.

Jurisdiction Assignment

Congress.gov bills are always assigned federal. Federal Register and FinCEN items are always federal. Overton papers default to international regardless of their geographic focus, because Overton's metadata does not include a reliable jurisdiction field. Manual entries can be assigned any jurisdiction value including specific state names.

7. Seed Data & Provenance

The tracker launched with 257 manually curated entries imported from an Excel-based master tracker. The seed dataset includes:

  • 103 legislation and regulatory entries -- federal bills dating to the early 2000s, including major statutes (TVPA, VAWA reauthorizations, FOSTA-SESTA, the Uyghur Forced Labor Prevention Act), key regulatory actions from HHS, DOJ, DHS, and DOL, and FinCEN advisories.
  • 40 lawsuit entries -- primarily civil actions under 18 U.S.C. 1595 against hotel chains, online platforms, and financial institutions, as well as selected DOJ criminal prosecutions. Case details were compiled from PACER filings, court opinions, and press releases.
  • 114 research entries -- peer-reviewed articles, systematic reviews, and policy reports, with particular coverage of survivor outcomes research, program evaluations, and anti-trafficking policy analysis.

The seed entries were hand-researched and individually compiled by the project lead through web scraping, document parsing, and direct review of court filings, federal databases, and academic sources. Data was organized and categorized in a structured Excel workbook before being converted to the JSON schema used here. Dates were normalized to ISO 8601 format using a multi-format parser; entries with no parseable date were assigned January 1 of the most recent year found in the text.

Entry selection was guided by focus questions centered on long-term survivor outcomes: which laws, cases, and studies account for well-being beyond initial service contact, and how the legal and research landscape measures or fails to measure survivor-defined success over time. This framing shaped what was included, particularly in the research corpus. Coverage is not exhaustive and reflects those particular priorities. Significant gaps remain in state-level legislation outside major trafficking states, non-English-language research, and cases resolved at the state court level.

8. Using the Interface

Tabs

The tab row at the top of the tracker filters entries by type. All displays the full combined dataset. Individual tabs (Legislation, Regulatory, Lawsuits, Research) filter to that entry type. The Digest tab is a special view -- see below.

Views

Three display modes are available for browsable tabs (not Digest):

  • List -- tabular display with sortable columns. Best for scanning many entries quickly or comparing status across items.
  • Cards -- card grid with title, type badge, status, date, and truncated summary. Best for reading entry-by-entry.
  • Timeline -- entries plotted chronologically by latest_action_date, grouped by year-month. Best for understanding how a policy area has evolved over time.

Digest Tab

The Digest tab is a two-column summary view. The left column shows entries updated within the last 30 days, grouped into "This Week" and "Earlier This Month." The right column shows all future scheduled events from the upcoming field across all entries, sorted by days until event. Days are color-coded: red (7 days or fewer), amber (8-30 days), grey (more than 30 days). The Digest tab respects the active search query and starred filter -- if starred is enabled, only starred items appear in both columns.

Detail Panel

Clicking any entry opens a detail panel on the right side. The panel shows all fields for the entry, organized by type-specific sections ("Policy Details," "Case Details," or "Research Details"), followed by the full update timeline (see Section 10). The panel can be closed with the [X] button or by clicking another entry. On narrow screens the panel opens as a full-screen overlay.

SYNC Button

The SYNC button in the header reloads all four data JSON files from the server and updates the in-memory dataset without a full page reload. Use it after pushing a data update to see the latest entries immediately. The button shows "SYNCING..." during the fetch and "OK" or "ERR" on completion. The "Last synced" timestamp reflects when the data was last loaded from the server.

Submit Entry Button

Opens a modal explaining the community submission process and linking to the GitHub Issue form. See Section 11.

The star toggle (accessible from the header) filters the full dataset to only entries you have starred. Stars are stored in your browser only and are never visible to others.

The Reset button clears all filter fields and returns to the default view (all entries, no filters). It does not affect the active tab or view mode.

10. Timeline & Update History

Each entry carries an updates array that records the history of significant actions, decisions, or publications associated with that item. The timeline is displayed in the detail panel, sorted chronologically. For legislation, it may include the introduction date, committee referrals, floor votes, and enactment. For lawsuits, it may include filing date, significant motions, summary judgment rulings, settlements, and verdicts. For research, it may include preprint posting, peer review, and publication.

Each update entry has the following structure:

  • date -- YYYY-MM-DD date of the event.
  • event -- Short description of what happened.
  • actor -- Person or institution responsible (sponsor, plaintiff, court, journal).
  • notes -- Optional additional context or analysis.
  • source_url -- URL of the source document for this event.

Automated fetches add new events to the timeline but never remove or edit existing ones. This means the full legislative, legal, or publication history accumulates over time as the tracker runs. Manually curated timelines in the seed dataset reflect research team curation and may not be complete for every entry.

11. Community Submissions

Any user with a GitHub account can submit a new entry to the tracker using the structured issue form accessed through the [+] Submit button. Submissions are processed automatically via a GitHub Actions workflow and appear on the live site shortly after submission.

The submission form collects: Entry Type, Title, Date, Summary (all required), plus optional fields for Bill/Case/DOI identifier, primary source URL, current status, trafficking types, jurisdiction, sponsor/author/plaintiff, and additional details. A confirmation checkbox requires the submitter to affirm that the entry is directly relevant to human trafficking policy, litigation, or research.

When the form is submitted, a GitHub Issue is opened with the title prefix [SUBMISSION]. The Actions workflow parses the structured form body using a regex split on the GitHub issue form section headers (### Field Label format), builds a JSON entry using the same schema as manually entered items, and appends it to the appropriate data file. A bot comments on the issue confirming success (or describing the error) and closes the issue.

Community-submitted entries are distinguished by a [C] community badge in the list and detail panel and carry a source value of "Community Submission (Issue #N)." The submitter's GitHub username appears in the submitted_by field. Lab team members can identify and review these entries using the badge or by filtering on the source field. If a submission needs to be removed, it can be deleted from the relevant data JSON file and the change committed and pushed.

12. Stars, Notes & Privacy

The tracker provides three private annotation tools: stars (bookmark an entry), notes (free-text annotation attached to an entry), and a general memo pad. All three are stored exclusively in the user's browser via the Web Storage API (localStorage). They are never transmitted to any server, never included in any network request, and are not visible to any other user.

Annotations persist as long as the user does not clear their browser storage for the site's domain. They are not synchronized across devices or browsers. Clearing browser data, using a private/incognito window, or accessing the tracker from a different browser will result in annotations not being visible. There is currently no export or import mechanism for annotations.

13. Known Limitations

  • Keyword-only title matching for Congress.gov. Bills are filtered by keyword presence in the bill title only. Bills with indirect relevance to trafficking (e.g., an appropriations bill that funds OVC programs) will not be captured unless their title contains a recognized keyword. Full-text search of bill content is not currently implemented.
  • No uniform state-level data source. State legislation must be added manually. Coverage of state law is uneven and reflects the research team's prior tracking activities. States with higher known trafficking case loads (CA, TX, FL) are likely better represented than others.
  • PACER access for lawsuits. Federal court docket information is not publicly available via API. Lawsuit entries depend on manual research and may lag behind actual case developments. Settlement amounts and verdict details are often unavailable due to confidentiality agreements.
  • Overton institutional access required. If the institutional subscription lapses or the API key expires, the automated research fetch will stop without error. Existing research entries are never deleted, but new publications will not be added automatically.
  • Outcome frame and evidence strength are manually assigned. Auto-fetched entries from Overton are not automatically classified. These fields must be manually reviewed and populated. Users filtering on these fields should be aware that entries without a value will be excluded from results, even if they are relevant.
  • Status mapping is approximate. The Congress.gov status mapping uses keyword matching on action text and does not capture all legislative states. Bills that were passed by one chamber but have not received action in the other, for example, are not distinguished from bills still in their originating chamber's committee.
  • Date normalization fallbacks. Entries with non-standard or ambiguous date strings are normalized using a series of fallback parsers. If no format matches, a year-only year receives a January 1 date, and a completely unparseable date is assigned today's date. Entries normalized by fallback should be manually verified.
  • Community submissions are not pre-screened. Entries submitted via GitHub Issues are added automatically without prior review. The confirmation checkbox is not verified for accuracy. Lab team members should periodically review community entries using the [C] badge filter to ensure quality and relevance.
  • Annotations do not persist across browsers or devices. Stars and notes are browser-local. There is no cloud sync or export.
