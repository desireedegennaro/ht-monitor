1. What This Is

This is a database of federal legislation, civil and criminal cases, and peer-reviewed research related to human trafficking. It's built for researchers, policy staff, and advocates who normally have to check Congress.gov, PACER, academic databases, and federal agency sites separately to get a complete picture. This puts it in one place.

It starts with 257 hand-built entries and adds new ones automatically every day by pulling from government APIs, RSS feeds, and research databases. Everything runs in your browser — there's no server, no login, and no tracking. Your annotations (stars, notes) stay on your device only.

The four categories:

  • Legislation — federal bills, enacted laws, and state statutes.
  • Regulatory — agency rules, guidance documents, and financial crime advisories. These are kept separate from legislation because they go through a different process and have different legal weight.
  • Lawsuits — civil suits and criminal cases, with a focus on civil actions under 18 U.S.C. § 1595 (the TVPA's beneficiary liability provision) and hotel/platform litigation.
  • Research — peer-reviewed articles, systematic reviews, and policy reports on interventions, survivor outcomes, and program evaluation.

2. How It's Built

The whole app is one HTML file (index.html) with CSS and JavaScript inside it. When you open the page, it loads four JSON data files from the same folder. JSON is just structured text — think of it as a spreadsheet that code can read. There's no database server, no backend, nothing to install. GitHub Pages hosts the files and serves them directly to your browser.

FileWhat's in itHow it gets updated
data/legislation.jsonBills and regulatory entriesDaily auto-fetch + manual
data/lawsuits.jsonCivil and criminal casesManual + community submissions
data/research.jsonResearch papers and reportsDaily auto-fetch + manual
data/meta.jsonEntry counts and fetch timestampsRewritten every fetch run
data/digest.jsonNew and updated items from the last runRewritten every fetch run

Every day at 8 AM EST, a GitHub Actions workflow (basically a scheduled script that runs in the cloud) calls government websites and research databases, processes the results, updates the JSON files, and pushes the changes to GitHub. GitHub Pages then serves the updated files. Your browser picks them up the next time you load the page or hit SYNC.

Stars and notes are stored in your browser's local storage — a small private storage space every browser keeps per website. They never leave your device.

3. Data Sources

Four sources are automated. Two are manual only.

Congress.gov API

The official API for U.S. federal legislation. It returns bill titles, sponsors, cosponsor counts, committee assignments, and the text of the most recent floor or committee action. Access requires a free API key (takes about 2 minutes at api.congress.gov). The fetch script sends 10 different search queries (e.g. "human trafficking," "uyghur forced labor," "survivor services trafficking") and pulls up to 25 results each. Each result is then checked against a keyword list before being added.

Federal Register API

The Federal Register is the U.S. government's official daily journal — it's where agencies publish final rules, proposed rules, and notices. The API is public and requires no key. The script runs 6 search queries (e.g. "human trafficking," "forced labor") and checks results against a keyword list. Everything pulled from here goes in as type regulatory, not legislation, because rules have a different legal pathway than bills.

Government RSS Feeds

RSS is a format websites use to broadcast their latest posts in a machine-readable list — like a news feed your code can subscribe to. The script monitors 6 feeds: DOJ press releases, HHS press releases, DHS news, FinCEN advisories (2 feeds), and State Dept press releases. Every item that mentions human trafficking-specific language is pulled in. See Section 4 for how the script filters out drug trafficking and other false matches.

PubMed and Semantic Scholar

PubMed is the National Library of Medicine's database of biomedical and public health research. Semantic Scholar is a broader academic search engine that covers social science, policy, and grey literature. Both are free, no key required. The script runs 10 PubMed queries and 5 Semantic Scholar queries. The same paper appearing in both databases is caught by the deduplication system (matched by DOI or PubMed ID) and stored as one entry, not two.

Overton

Overton indexes policy papers and reports alongside academic journals. It requires an institutional subscription. Northeastern has access — the key goes in GitHub's secrets so the daily script can use it without it ever being visible in the code. If the key is missing, the script skips Overton and uses PubMed and Semantic Scholar instead.

PACER and State Legislatures

PACER is the federal court filing system. There's no public API that lets code pull cases automatically, so all lawsuit entries are added by hand. State legislation has the same problem — each state has its own system, no uniform national API. State entries are added manually.

4. How the Daily Fetch Works

The script runs daily and looks back 90 days by default. The challenge it solves: the same bill or paper shows up across multiple search queries, and the same RSS item appears in the feed for days in a row. Without a deduplication system, every run would add duplicates. With a naive overwrite, any notes or classifications you've added would get wiped. The pipeline handles both problems.

Accuracy filtering for RSS

Government RSS feeds are noisy. DOJ publishes drug trafficking prosecutions every day. "Forced labor" shows up in wage theft cases. Before any RSS item gets accepted, it has to pass two checks:

  • Layer 1 — must contain a specific phrase. Single words like "trafficking" or "exploitation" aren't enough. The item has to contain a compound phrase like "human trafficking," "sex trafficking," "forced labor," "debt bondage," or a specific acronym like "TVPA" or "UFLPA."
  • Layer 2 — false positive check. Even after passing layer 1, the text is scanned for disqualifying context. If it contains "human trafficking" or "sex trafficking" or "forced labor" → accepted immediately. If it contains "drug trafficking," "fentanyl trafficking," "narcotics trafficking," "weapons trafficking," etc. without any of those confirming phrases → rejected. A DOJ press release about a fentanyl conviction that mentions TVPA in passing fails here. A real human trafficking prosecution passes immediately.

Deduplication

Before processing anything, the script builds an index of everything already in the database, organized by five different lookup keys:

  1. Item ID — catches the same bill coming back from two different search queries in the same run
  2. DOI (the permanent ID for research papers) — catches the same paper from PubMed and Semantic Scholar
  3. PubMed ID — backup cross-source match for research
  4. URL — catches the same RSS item seen on multiple days
  5. Bill number + Congress (e.g. HR1234-119) — catches the same bill when the ID was formatted differently on different fetches

Each incoming item is checked against all five keys. If any one matches, the item already exists.

Merge logic

Once the script knows whether an item is new or already exists, it does one of three things:

  • New item — adds it to the database and indexes it
  • Existing item with changes — updates the status and latest action in the existing record, appends the new event to its timeline, and flags it for the digest. Nothing else changes.
  • Existing item, no changes — skips it

Fields you fill in manually — notes, evidence strength, outcome frame, relevance, your position, funder, organizations — are never touched by the auto-fetch. They survive every run.

Digest

After each run, the script writes data/digest.json with two lists: genuinely new items added this run, and existing items that got a status or action update. The Digest tab reads from this file. Each card shows what changed — for updates, it includes a plain-language summary like "In Committee → Passed House" and the specific new event.

6. How Things Get Classified

Trafficking Type

For auto-fetched entries, the script guesses trafficking type from words in the title and abstract. It checks for sex trafficking terms ("sex trafficking," "prostitution," "csec," "commercial sex") and labor trafficking terms ("labor trafficking," "forced labor," "debt bondage," "supply chain"). If neither set matches, it defaults to both ["sex", "labor"] — general anti-trafficking bills like TVPA reauthorizations usually cover both. This is a guess. If the classification is wrong for a specific entry, it can be corrected manually in the data file.

Bill Status

Congress.gov returns the text of the most recent action, not a status code. The script maps that text to one of five values:

If the action text says…Status assigned
"signed by president" or "became public law"Enacted
"passed senate"Passed Senate
"passed house"Passed House
"committee," "subcommittee," or "hearing"In Committee
anything elseIntroduced

This is approximate. It won't catch every possible legislative state — in particular, it doesn't distinguish between a bill that died in committee years ago and one that just got referred. For anything that needs precision, check the linked Congress.gov page directly.

Outcome Frame

The outcome_frame field tracks how a law or study defines success — specifically whether it measures service delivery (beds filled, intakes processed, referrals made) or actual survivor well-being over time. This field was added to support research on the gap between those two things. It's not pulled from any external source — it has to be manually assigned. Values:

  • Service Metrics Only — counts services delivered, no survivor-reported outcomes
  • Survivor: Short-Term (0–6 months)
  • Survivor: Medium-Term (6–12 months)
  • Survivor: Long-Term (>12 months)
  • Mixed / Both

Evidence Strength

Applied to research entries. Manually assigned — not calculated automatically. Values: Strong, Moderate, Limited, Descriptive (for qualitative work that's valuable but not designed to prove efficacy).

Case Category

Lawsuit entries get a case_category tag based on the sector named as defendant: HOSPITALITY INDUSTRY, ONLINE PLATFORMS, FINANCIAL, LABOR / EMPLOYMENT, GOVERNMENT, INDIVIDUAL. This is for filtering by industry, not a legal determination about the defendant.

Policy Mechanism

Manually assigned for legislation. Describes the main tool the law uses: Service Funding, Criminalization, Platform Liability, Supply Chain, Victim Identification, Survivor Compensation, Cross-Sector Coordination. One value per entry even if a bill addresses multiple mechanisms.

7. Where the Starting Data Came From

The tracker launched with 257 entries built by hand. These came from an Excel-based tracker that was converted into the JSON format used here. The starting set includes:

  • 103 legislation and regulatory entries — federal bills going back to the early 2000s, including TVPA, VAWA reauthorizations, FOSTA-SESTA, UFLPA, key HHS and DOJ rules, and FinCEN advisories.
  • 40 lawsuit entries — mostly civil actions under 18 U.S.C. § 1595 against hotel chains, online platforms, and financial institutions, plus selected DOJ criminal prosecutions. Details came from PACER filings, court opinions, and press releases.
  • 114 research entries — peer-reviewed articles, systematic reviews, and policy reports, focused especially on survivor outcomes research and program evaluations.

The starting entries were selected with specific research questions in mind: which laws, cases, and studies account for long-term survivor well-being, and how does the legal and research landscape measure (or fail to measure) outcomes beyond initial service contact. The dataset reflects those priorities — it's not meant to be exhaustive. State law coverage is uneven, non-English research is largely absent, and state-court cases are mostly missing.

8. Using the Interface

Tabs

All shows every entry. The other tabs (Legislation, Regulatory, Lawsuits, Research) filter by type. Digest is its own view — see below.

Views

  • List — table with sortable columns. Good for scanning lots of entries.
  • Cards — cards with title, status, date, and summary. Good for reading entry by entry.
  • Timeline — entries plotted chronologically by most recent action date, grouped by month. Good for seeing how a policy area has changed over time.

Digest Tab

Two columns. Left: entries updated in the last 30 days, split into "This Week" and "Earlier This Month." Right: upcoming scheduled events from the upcoming field across all entries, sorted by how soon they are. Days are color-coded: red (7 days or fewer), amber (8–30 days), grey (more than 30). The Digest respects the search bar and starred filter.

Detail Panel

Click any entry to open its detail panel on the right. Shows all fields organized by type, plus the full update timeline. Close it with [X] or click another entry. On small screens it opens full-screen.

SYNC

The SYNC button reloads the data files from GitHub without refreshing the page. Use it if you've just pushed an update and want to see it immediately. Shows "SYNCING…" while loading and the timestamp of the last successful load.

Submit Entry

Opens a modal with instructions and a link to the GitHub Issue form for submitting new entries. See Section 11.

10. Update Timelines

Every entry has an updates array — a running log of significant actions, decisions, or publications. For bills: introduction, committee referrals, votes, enactment. For lawsuits: filing, key motions, rulings, settlements. For research: preprint, publication, citations.

Each event in the timeline has: date, event (what happened), actor (who did it), notes (optional context), and source_url. The daily fetch adds new events but never removes or edits existing ones, so the full history accumulates over time.

11. Submitting Entries

Anyone with a GitHub account can submit a new entry using the form linked from the [+] Submit button. Required fields: Entry Type, Title, Date, Summary, URL. Optional: identifier, status, trafficking types, jurisdiction, sponsor/author/plaintiff, and details.

When you submit, a GitHub Issue opens. An automated script reads the form, builds a JSON entry using the same structure as everything else in the database, and adds it to the right data file. A bot comments on the issue confirming it worked (or explaining what went wrong) and closes the issue. The entry shows up on the site within a few minutes.

Community-submitted entries show a [C] community badge and list their source as "Community Submission (Issue #N)." To remove a bad submission, delete it from the JSON file and push the change.

12. Stars, Notes & Privacy

Stars, notes, and the memo pad are stored in your browser's local storage — a small private space each website gets on your device. Nothing is sent anywhere. Other users can't see your annotations.

Annotations stay until you clear your browser data for this site. They don't sync across devices or browsers. There's no export or import right now — if you switch browsers or clear your storage, they're gone.

13. Known Gaps

  • Congress.gov filtering is title-only. A bill has to mention a recognized keyword in its title to be pulled in. Bills that fund anti-trafficking programs without naming them explicitly in the title won't be caught. Full-text bill search isn't implemented.
  • State legislation is mostly manual. No uniform national API exists for state legislative data. Coverage reflects what was already in the Excel tracker before launch and what gets manually added. States with higher known case volumes are probably better covered than others.
  • Lawsuit data lags real developments. Federal court filings aren't publicly accessible via API. Entries depend on manual research and may be months behind actual case status. Settlement amounts are often confidential and missing.
  • Outcome frame and evidence strength must be filled in manually. These fields are blank for all auto-fetched entries until someone reviews them. Filtering on these fields excludes entries where they haven't been set yet, even if those entries are relevant.
  • Status mapping is approximate. The bill status assigned by the script is based on keyword matching in the action text. It doesn't distinguish between a bill that died in committee three years ago and one that was just referred. Check the Congress.gov link for anything status-sensitive.
  • Overton requires an active institutional key. If the key expires, the research auto-fetch stops quietly. PubMed and Semantic Scholar keep running. Existing entries are never deleted.
  • Annotations don't sync across devices. Stars and notes are browser-local. No cloud backup, no export.
  • Community submissions aren't reviewed before they go live. The confirmation checkbox isn't verified. Lab team members should periodically check for bad entries using the [C] badge filter.
